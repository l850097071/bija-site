AutoDFT：让 AI Agent 替你跑 VASP 计算

2026-05-30 Agent 系统 · DFT 计算

如果你做过 DFT 计算，你一定经历过这些：输入文件里少写了一个参数、INCAR 标签拼错、提交脚本路径不对、服务器排队排到天亮后发现收敛失败——然后重来。

清华团队 2026 年 5 月发布的 AutoDFT 就是来解决这件事的。一个多 Agent 的闭环 DFT 框架，用大模型驱动，从输入文件生成到失败诊断到自动修复——全自动跑通。在包含 34 个任务、9 种 DFT 计算类型的 VASPBench 上，成功率 94.1%。

它不是"又一个 DFT 脚本"

传统的 DFT 自动化脚本做的事是：给定输入文件 → 提交 → 检查输出 → 如果失败就重试（用同样的参数）。这是"盲跑"。

AutoDFT 不同：

  1. 规划阶段：一个 Planner Agent 解读你的任务（"我想算这个体系的吸附能"），然后生成完整的计算方案——选什么泛函、用什么赝势、K 点怎么设。
  2. 执行阶段：提交 VASP，监控作业状态，解析输出。
  3. 诊断阶段：如果计算失败或不合理，一个 Diagnoser Agent 分析输出文件——是 SCF 不收敛？是结构不合理？是参数冲突？
  4. 修复阶段：根据诊断结果自动修改输入参数——调整 mixing、改 smearing、换算法——然后重新提交。
  5. 迭代：重复 2-4 直到收敛或确认该计算路径不可行。
关键区别：AutoDFT 不是在盲跑——它是在"理解失败原因并修正"。这是 Agent 和脚本的本质差异。

94.1% 意味着什么

VASPBench 包含 34 个精心设计的任务，覆盖 9 种 DFT 计算类型：结构优化、态密度、能带、功函数、光学性质、弹性常数、声子、Bader 电荷、STM 模拟。这些不是玩具——是真实科研中每天在跑的东西。

94.1% 的成功率意味着 34 个任务中大约 32 个能自动跑通。作为对比，一个没有 DFT 经验的实验组研究生，在同样的任务集上——跟着教程做——成功率大概不会超过 50%。

论文测试了 GPT-5.2、Claude 4 Sonnet 等多个模型。值得注意的是，越强的模型，诊断和修复能力越强——这意味着随着底层模型继续迭代，AutoDFT 的成功率还会上升。

对科研的真正意义

不要只看"省时间"。看待 AutoDFT 的正确方式是：

它让非 DFT 专家也能跑可靠的第一性原理计算。

一个做材料合成的实验组，不需要招一个 DFT 博士后。他们只需要描述自己想算什么，AutoDFT 处理剩下的。这不是"降低科研门槛"——这是让更多科学家能用上第一性原理工具来验证他们的假设。

反过来，对 DFT 专家来说，AutoDFT 的价值是把时间从"调参数"转移到"想问题"。你不再花一下午调 INCAR——你花一下午想"该算什么、为什么"。这才是研究者该做的事。

跟 Bija 的关系

AutoDFT 是多 Agent 闭环——和 Bija 的 DSEIM（发现-搜索-评估-集成-度量）自进化循环是同构的。两者的核心逻辑一样：

  • 感知失败 → 定位原因 → 修正策略 → 重试 → 从经验中学习

区别在于：AutoDFT 专注 DFT 计算这一件事。Bija 想做的是把这个循环应用到任何认知工作——包括做 DFT 的 Agent 本身的自我改进。

你可以把 AutoDFT 看成 Bija 的一个"技能模块"。或者说，Bija 是一个会自我进化的元 Agent，而 AutoDFT 是它能调用的一种具体能力。这两者不是竞争关系——是垂直 Agent × 通用进化框架的互补。

现在还缺什么

AutoDFT 的代码已经开源（GitHub: thu-nmrc）。但目前有三个明显的局限：

  1. 只支持 VASP——QE、CP2K、LAMMPS 用户暂时只能等。
  2. 需要付费模型——GPT-5.2 不便宜。虽然可以用 Claude Haiku 等低成本模型，但成功率会降。
  3. 没有到"全自主"——94.1% 的意思是有 5.9% 的任务仍然需要人类介入。在关键的科研计算中，这可能意味着你需要一个懂 DFT 的人 standby。

但这些都不是根本性问题。根本性问题是：当 DFT 计算的执行层被 Agent 自动化后，你的价值在哪里？

不是"会跑 VASP"。是知道该算什么、为什么、以及结果意味着什么。这是人类研究者的不可替代性——不是执行计算，是做科学判断。科研品味。
